# Neutral Naming Guide

Use this when the user wants a public repository, package, command, or skill name.

## Naming principles

- Prefer invented or descriptive names that do not imply affiliation with another product.
- Avoid third-party product names in the repo title, skill name, command names, badges, screenshots, and logo.
- Avoid similar slogans, mascots, or visual identity.
- Keep compatibility notes factual and secondary.
- Search before shipping a name if the project may become commercial.

## Safe name directions

| Direction | Examples |
|---|---|
| Workflow/rigging | `flowrig`, `repo-rig`, `task-rig` |
| Shipping loop | `shiploop`, `release-loop`, `patchloop` |
| Review deck | `patchdeck`, `diffdeck`, `reviewdeck` |
| Gates/checks | `shipgate`, `mergegate`, `testgate` |
| Context control | `context-rail`, `scope-rail`, `work-rail` |

## Names to avoid

- Product/vendor names unless the vendor explicitly allows it.
- Lookalike names that create confusion.
- Names that imply certification, partnership, or official status.
- Repo descriptions that say “the official alternative to X”.

## Recommended metadata wording

Use:

> A Codex skill for gated repository work: audit, plan, patch, verify, and handoff.

Avoid:

> Best practices for [vendor/product] copied into Codex.
